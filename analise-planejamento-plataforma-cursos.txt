================================================================================
O QUE FOI ENTENDIDO
================================================================================

É o planejamento de uma plataforma proprietária de cursos online premium (ticket
R$18k/ano) com dois diferenciais centrais:

1. Privacy by Design — zero dados pessoais no banco próprio, tudo fica na Stripe.
2. Identidade por user_token — não por e-mail.

O documento tem 9 partes integradas, 66 páginas, 11 tabelas de banco de dados e
passou por 6 versões com 3 rodadas de autocrítica. A arquitetura central é sólida
e o problema de negócio (duplicação de contas, compartilhamento, falta de
anonimato em cursos premium) é real e bem quantificado.

================================================================================
O QUE ESTÁ ERRADO (PROBLEMAS SÉRIOS)
================================================================================

1. device_token em localStorage = VULNERABILIDADE XSS
---
O mecanismo primário de anti-sharing usa um token armazenado em localStorage.
Qualquer script injetado via XSS consegue roubar esse token — e com ele, o
acesso ao curso. Para um produto de R$18k/ano isso é inaceitável.

Solução correta: httpOnly cookie com SameSite=Strict e Secure. O documento não
menciona isso em nenhuma das 66 páginas.

2. JWT + tabela sessions = PARADOXO ARQUITETURAL
---
O campo session_token (VARCHAR 128) na tabela sessions armazena um JWT, mas o
mesmo registro tem is_active BOOLEAN para invalidação server-side.

Isso mistura dois modelos incompatíveis:
- JWT é stateless: não precisa de banco, não pode ser invalidado facilmente.
- Opaque token é stateful: precisa de banco, pode ser invalidado a qualquer hora.

Se você precisa de is_active (e precisa, para sessão única), use opaque token —
não JWT. Quem for implementar vai tomar uma decisão arbitrária nesse ponto e
provavelmente errar.

3. CRONOGRAMA DE 12 SEMANAS COM 1-2 DEVS = IRREALISTA
---
O escopo cobre:
- Stripe webhooks com idempotência
- Sistema de identidade com 4 sinais e 3 estados de resolução
- Autenticação PIN com proteção anti-enumeração e rate limiting
- Magic links por e-mail
- Player de vídeo com progresso
- Anti-sharing (device token, sessão única, PIN)
- Painel admin completo
- Sistema de migração com claim codes e legacy_payments
- Backup com RPO de 5 minutos

Isso é 6+ meses para uma equipe sênior bem entrosada. O Gantt coloca
"Testes + deploy" apenas nas semanas 11-12. Testar um sistema de pagamentos
só no final é o caminho direto para bugs em produção com dinheiro real.

4. CÁLCULO DE PAYBACK É INGÊNUO
---
O documento afirma payback de 3 meses baseado em recuperar R$360k/ano de
compartilhamento. Isso pressupõe que 100% dos usuários que compartilhavam login
vão virar compradores pagantes após a implementação.

Na realidade, a conversão de "usuário que usava login emprestado" para "cliente
pagante" é tipicamente 10 a 20%. O payback real, com premissas conservadoras
honestas, é de 12 a 18 meses. Apresentar 3 meses para um cliente grande é
arriscado — quando o número não se concretizar, afeta a credibilidade do projeto.

5. OS "4 SINAIS" NUNCA SÃO EXPLICITAMENTE LISTADOS
---
O conceito central do sistema — a vinculação por 4 sinais com 3 estados — é
referenciado dezenas de vezes ao longo do documento mas os 4 sinais nunca são
enumerados claramente num único lugar.

Dá para inferir que são: e-mail, stripe_customer_id, fingerprint do cartão
(card_last4 + bin), device token. Mas um documento v6.0 FINAL não pode ter sua
feature core ambígua. Qualquer dev novo no projeto vai ter dúvida aqui.

================================================================================
O QUE PRECISA MELHORAR
================================================================================

TÉCNICO
-------

- Mux OU Bunny Stream: decisão em aberto num documento "final". São arquiteturas
  diferentes — CDN próprio vs. plataforma gerenciada, preços distintos, APIs
  distintas. Precisa de escolha antes de qualquer implementação.

- Sem estratégia de retry/dead-letter para webhooks com falha persistente.
  A tabela stripe_events garante idempotência, mas se o handler falhar repetidas
  vezes (banco indisponível, bug de código), o evento some silenciosamente.
  Precisa de fila com retry e alerta para falhas persistentes.

- Rate limiting só especificado para PIN. Faltam os endpoints de: checkout,
  resgate de claim codes, painel admin, e signed URLs de vídeo.

- Fluxo de recuperação quando usuário limpa localStorage não está documentado.
  O usuário perde o device_token, não consegue mais logar pelo device, e o
  mecanismo de recuperação nesse caso específico não está descrito.

LGPD
----

O argumento "banco comprometido = zero dados pessoais" é forte, mas incompleto.
Os seguintes dados no banco próprio são dados pessoais sob a LGPD:
- stripe_customer_id (identificador vinculado a uma pessoa real)
- card_last4 + bin (informação financeira pessoal)
- Dados comportamentais: IPs, dispositivos, progresso, timestamps

O documento não define:
- Responsável pelo tratamento (controlador vs. operador)
- Base legal para cada categoria de dado (legítimo interesse, contrato, etc.)
- Política de retenção de dados (quanto tempo ficam os logs de sessão?)
- Procedimento de notificação de incidente (ANPD exige 72h)

PLANEJAMENTO
------------

- Railway como host de produção para um produto de R$18k/comprador não tem SLA
  definido no documento. Para esse nível de ticket, o cliente produtor vai exigir
  garantia de uptime. Railway não oferece SLA contratual. Para produção com esse
  volume financeiro, considerar AWS/GCP com RDS e load balancer.

- A fase multi-tenant (meses 4-8) é tratada em 3 linhas. Stripe Connect é uma
  mudança arquitetural profunda — não é uma feature, é uma reescrita parcial do
  sistema de pagamentos e isolamento de dados. Precisa de documento próprio.

- Testes unitários e de integração não aparecem como parte do desenvolvimento —
  só como atividade final (semanas 11-12). Para o sistema de identidade, que é
  o coração do produto, TDD seria a abordagem correta: escrever os testes dos
  4 sinais antes de implementar a lógica de vinculação.

DOCUMENTAÇÃO
------------

- A afirmação "0 lacunas conhecidas" reduz a credibilidade do documento com
  revisores técnicos externos. Todo sistema complexo tem lacunas; a diferença é
  entre lacunas conhecidas e desconhecidas. Essa afirmação soa como marketing
  interno, não como engenharia honesta.

- As partes II a IX são resumos que apontam para "PDFs individuais por PDF_ID".
  Se o objetivo é entregar um documento completo e autossuficiente para o cliente,
  as seções precisam ter o conteúdo completo — não referências a outros arquivos
  que o cliente não necessariamente tem acesso.

================================================================================
VEREDICTO GERAL
================================================================================

O problema de negócio é real, o diferencial é válido e a arquitetura central
(user_token + Privacy by Design) é sólida. A evolução documentada de v1 a v6
demonstra pensamento crítico iterativo — isso é um ponto forte.

Mas o documento está superestimado em 3 frentes:

  TIMELINE   → 12 semanas é marketing. Seja honesto: 20-24 semanas.
  PAYBACK    → 3 meses é otimismo. Apresente 3 cenários (conservador/base/otimista).
  COMPLETUDE → "0 lacunas" é arrogância técnica. Documente as incertezas conhecidas.

Antes de apresentar para o cliente, as correções prioritárias são:
  1. Resolver o paradoxo JWT/sessions (decidir: stateless ou stateful)
  2. Substituir localStorage por httpOnly cookie para device_token
  3. Enumerar explicitamente os 4 sinais de vinculação em um único lugar
  4. Revisar o cronograma com estimativas honestas
  5. Apresentar payback em 3 cenários com premissas explícitas
